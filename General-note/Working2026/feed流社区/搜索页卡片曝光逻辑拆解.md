## 二、核心数据结构

  

### 2.1 Ref 变量说明

  

| 变量名 | 类型 | 作用 |

|--------|------|------|

| `isScrollToBottom` | `Ref<boolean>` | 当前滚动方向，true=向下，false=向上 |

| `lastScrollTop` | `Ref<number>` | 上次滚动位置，用于计算滚动方向 |

| `reportedIndexes` | `Ref<Record<number, boolean>>` | 已上报卡片缓存，key=卡片index，value=上报时的滚动方向 |

| `unreportedItems` | `Ref<ReportItem[]>` | 本次待上报的曝光项列表 |

| `cardTimeoutID` | `Ref<Record<number, NodeJS.Timeout>>` | 每个卡片的定时器ID缓存 |

  

### 2.2 常量

  

| 常量名 | 值 | 含义 |

|--------|-----|------|

| `cardTimeout` | `1000` | 有效曝光时间阈值（毫秒） |

| `threshold` | `[0, 0.5]` | IntersectionObserver 触发阈值 |

| `rootMargin` | `"-60px 0px 0px 0px"` | 视口顶部偏移（导航栏高度） |

## 三、模块详解

  

### 3.1 滚动方向检测

  

```typescript

const onScrollChange = (scrollTop: number) => {

// 防止手抖导致误判，40px 为经验阈值

isScrollToBottom.current =

scrollTop - lastScrollTop.current > 0 ||

(isScrollToBottom.current && lastScrollTop.current - scrollTop < 40);

  

lastScrollTop.current = scrollTop;

};

```

  

#### 流程示例

  

```

初始状态: lastScrollTop = 0, isScrollToBottom = true

  

操作1: 向下滚动到 500px

scrollTop = 500

500 - 0 = 500 > 0 → isScrollToBottom = true ✅

更新: lastScrollTop = 500

  

操作2: 轻微向上滚动 20px（手抖）

scrollTop = 480

480 - 500 = -20 < 0，但 500 - 480 = 20 < 40

→ 保持 isScrollToBottom = true（不误判）✅

更新: lastScrollTop = 480

  

操作3: 明确向上滚动 100px

scrollTop = 380

380 - 480 = -100 < 0，且 480 - 380 = 100 > 40

→ isScrollToBottom = false ✅

更新: lastScrollTop = 380

```

### 3.3 定时器机制（有效曝光判定）

  

#### 核心逻辑

  

```tsx

onChange={(inView, entry) => {

// 情况A: 卡片可见 ≥ 50%，且没有正在计时

if (inView && entry.intersectionRatio >= 0.5 && !cardTimeoutID[props.index]) {

let id = setTimeout(() => {

const _timeId = (entry.target as any).timeId;

// 验证：1秒后仍是同一次曝光

if (_timeId && _timeId === cardTimeoutID[props.index]) {

onCardDispose(props.data, props, entry.intersectionRatio);

cardTimeoutID[props.index] = null;

}

}, cardTimeout); // 1000ms

cardTimeoutID[props.index] = id;

(entry.target as any).timeId = id;

}

// 情况B: 卡片离开或可见 < 50%

else {

if (cardTimeoutID[props.index]) {

clearSingleTimeout(props.index);

delete (entry.target as any).timeId;

}

}

}}

```

  

#### 流程示例 1：正常曝光（持续可见 1 秒以上）

  

```

时间线:

0s 1s 2s

│ │ │

▼ ▼ ▼

卡片进入视口 定时器触发 持续可见

(ratio >= 50%) 上报曝光!

状态变化:

T=0s: cardTimeoutID[0] = 100, target.timeId = 100

T=1s: 检查 100 === 100 ✅ → 调用 onCardDispose()

cardTimeoutID[0] = null

  

结果: ✅ 上报 1 次曝光

```

  

#### 流程示例 2：快速划过（不足 1 秒）

  

```

时间线:

0s 0.5s 1s

│ │ │

▼ ▼ │

卡片进入 卡片离开 (定时器已取消)

  

状态变化:

T=0s: cardTimeoutID[0] = 100, target.timeId = 100

T=0.5s: clearTimeout(100), cardTimeoutID[0] = null, delete timeId

  

结果: ❌ 不上报（未达到 1 秒）

```

  

#### 流程示例 3：曝光 0.5s → 离开 → 回来曝光 2s

  

```

时间线:

0s 0.5s 1s 1.5s 2s 2.5s

│ │ │ │ │ │

▼ ▼ │ ▼ ▼ │

进入 离开 │ 进入 1秒到期 持续

│ 上报!

  

状态变化:

T=0s: cardTimeoutID[0] = 100, target.timeId = 100

T=0.5s: clearTimeout(100), cardTimeoutID[0] = null

T=1.5s: cardTimeoutID[0] = 200, target.timeId = 200

T=2.5s: 检查 200 === 200 ✅ → 上报!

  

结果: ✅ 上报 1 次曝光（第二次进入后满足条件）

```

  

#### 双重 ID 验证的意义

  

| 存储位置 | 用途 |

|----------|------|

| `cardTimeoutID[index]` | 集中管理所有定时器 |

| `target.timeId` | 定时器触发时验证是否是"同一次"曝光 |

  

**防止的问题：**

- DOM 元素被虚拟列表复用

- 快速进出产生的竞态条件

- Tab 切换时的残留定时器

### 3.4 曝光收集 (onCardDispose)

  

```typescript

const onCardDispose = (data, props, ratio) => {

if (data.isActivity) return; // 活动卡片不上报

  

unreportedItems.current.push({

uuid: data.uuid,

reportIndex: props.index,

score: data.score,

isOriginal: data.isOriginal,

displayOrder: data.displayOrder,

isTemplate: isTemplateItem(data),

_name: data.name,

_isScrollToBottom: isScrollToBottom.current, // 记录当时的滚动方向

});

  

sendReport.run(); // 触发防抖上报

};

```

  

#### 流程示例

  

```

滚动过程中，多个卡片触发曝光:

  

T=0ms: 卡片0 触发 onCardDispose → unreportedItems = [卡片0]

T=100ms: 卡片1 触发 onCardDispose → unreportedItems = [卡片0, 卡片1]

T=200ms: 卡片2 触发 onCardDispose → unreportedItems = [卡片0, 卡片1, 卡片2]

  

每次都调用 sendReport.run()，但由于防抖，实际执行延迟到最后一次调用的 500ms 后

```

### 3.5 防抖上报 (sendReport)

  

```typescript

const sendReport = useDebounceFn(

() => {

if (unreportedItems.current.length) {

// 1. 滚动方向变化时清理缓存

// 2. 去重过滤

// 3. 构建上报数据

// 4. 发送埋点

}

},

{ wait: 500 } // 防抖 500ms

);

```

  

#### 为什么用防抖？

  

| 不使用防抖 | 使用防抖 500ms |

|-----------|---------------|

| 每个卡片单独发请求 | 500ms 内的曝光合并为一次请求 |

| 快速滚动可能产生几十个请求 | 大幅减少请求数量 |

| 服务器压力大 | 批量上报，性能更好 |

  

#### 流程示例

  

```

时间线:

0ms 100ms 200ms 300ms 700ms

│ │ │ │ │

▼ ▼ ▼ │ ▼

卡片0 卡片1 卡片2 │ 实际执行

曝光 曝光 曝光 │ sendReport()

│ │ │ │ │

└────────┴────────┴───────┘ │

收集到 unreportedItems │

▼

批量上报 [卡片0, 卡片1, 卡片2]

```

### 3.6 去重逻辑（滚动方向感知）

  

#### 第一步：清理旧方向缓存

  

```typescript

Object.keys(_reportedIndexes).forEach((index) => {

// 如果滚动方向变化，清理之前方向的缓存

if (Reflect.has(_reportedIndexes, index) &&

_reportedIndexes[Number(index)] !== isScrollToBottom.current) {

delete _reportedIndexes[Number(index)];

}

});

```

  

#### 第二步：过滤已上报项

  

```typescript

let uniqueItems = _unreportedItems

.sort((a, b) => a.reportIndex - b.reportIndex)

.filter((item) => {

const cachedScroll = _reportedIndexes[item.reportIndex];

// 更新缓存

reportedIndexes.current[item.reportIndex] = item._isScrollToBottom;

// 未上报过 或 滚动方向不同 → 保留

return typeof cachedScroll === 'undefined' || cachedScroll !== item._isScrollToBottom;

});

```

  

#### 完整流程示例

  

**场景：向下滚动看完卡片 0-5，然后向上滚动回看卡片 3-1**

  

```

=== 阶段1：向下滚动，曝光卡片 0, 1, 2 ===

  

isScrollToBottom = true

  

sendReport 执行:

1. 清理缓存: reportedIndexes = {} (空，无需清理)

2. 过滤: 全部通过 → uniqueItems = [卡片0, 卡片1, 卡片2]

3. 更新缓存: reportedIndexes = { 0: true, 1: true, 2: true }

4. 上报: [卡片0, 卡片1, 卡片2]

  

=== 阶段2：继续向下滚动，曝光卡片 3, 4, 5 ===

  

isScrollToBottom = true

  

sendReport 执行:

1. 清理缓存: 方向相同，无需清理

2. 过滤: 全部通过 → uniqueItems = [卡片3, 卡片4, 卡片5]

3. 更新缓存: reportedIndexes = { 0: true, 1: true, 2: true, 3: true, 4: true, 5: true }

4. 上报: [卡片3, 卡片4, 卡片5]

  

=== 阶段3：向上滚动，回看卡片 3, 2, 1 ===

  

isScrollToBottom = false (方向变了!)

  

sendReport 执行:

1. 清理缓存:

遍历 { 0: true, 1: true, 2: true, 3: true, 4: true, 5: true }

每个值都是 true，当前方向是 false

true !== false → 全部删除!

reportedIndexes = {}

  

2. 过滤:

缓存已清空，全部通过 → uniqueItems = [卡片1, 卡片2, 卡片3]

  

3. 更新缓存: reportedIndexes = { 1: false, 2: false, 3: false }

  

4. 上报: [卡片1, 卡片2, 卡片3] (作为"向上滚动"的曝光)

  

=== 阶段4：继续向上滚动，卡片 1, 2 再次进入视口（但已上报过） ===

  

isScrollToBottom = false

  

sendReport 执行:

1. 清理缓存: 方向相同，无需清理

2. 过滤:

卡片1: reportedIndexes[1] = false, 当前 = false → 相同，过滤掉

卡片2: reportedIndexes[2] = false, 当前 = false → 相同，过滤掉

uniqueItems = [] (空)

  

3. 不上报（没有新曝光项）

```

  

#### 去重规则总结

  

| 情况 | 是否上报 |

|------|----------|

| 从未上报过 | ✅ 上报 |

| 上报过，但滚动方向不同 | ✅ 上报（算新的一次曝光） |

| 上报过，滚动方向相同 | ❌ 不上报（去重） |

### 3.7 清理机制

  

#### 3.7.1 清理单个定时器

  

```typescript

const clearSingleTimeout = useCallback((index: number) => {

if (cardTimeoutID.current[index]) {

clearTimeout(cardTimeoutID.current[index] as NodeJS.Timeout);

cardTimeoutID.current[index] = null;

}

}, []);

```

  

**触发时机：** 卡片离开视口或可见比例 < 50%

  

#### 3.7.2 批量清理所有定时器

  

```typescript

const cleanupTimeouts = useCallback(() => {

Object.values(cardTimeoutID.current).forEach((id) => {

if (id) clearTimeout(id);

});

cardTimeoutID.current = {};

}, []);

```

  

**触发时机：**

  

```typescript

// 1. 组件卸载时

useEffect(() => {

return () => {

cleanupTimeouts();

};

}, [cleanupTimeouts]);

  

// 2. 切换 Tab 时

useEffect(() => {

cleanupTimeouts();

}, [activeTab, cleanupTimeouts]);

```

  

#### 流程示例：Tab 切换

  

```

=== 用户在"模型"Tab，卡片 0, 1, 2 正在计时 ===

  

cardTimeoutID = { 0: 100, 1: 101, 2: 102 }

  

=== 用户切换到"图片"Tab ===

  

useEffect 触发 cleanupTimeouts():

1. clearTimeout(100)

2. clearTimeout(101)

3. clearTimeout(102)

4. cardTimeoutID = {}

  

结果: 旧 Tab 的定时器全部取消，不会产生错误的曝光上报

```

## 五、设计亮点总结

  

| 特性 | 实现方式 | 解决的问题 |

|------|----------|-----------|

| 有效曝光 | 1 秒定时器 | 过滤快速划过的无效曝光 |

| 防止重复上报 | reportedIndexes 缓存 | 同方向重复曝光不计数 |

| 滚动方向感知 | isScrollToBottom | 上下滚动分开计数 |

| 性能优化 | 防抖 500ms | 减少网络请求数量 |

| 内存安全 | cleanupTimeouts | 组件卸载/Tab切换时清理 |

| 竞态安全 | 双重 timeId 验证 | DOM 复用、快速进出场景 |

| 导航栏遮挡 | rootMargin | 被遮挡部分不算可见 |

| 手抖容错 | 40px 阈值 | 轻微回滚不改变方向判定 |
