## 一、Monorepo 和微前端的本质区别

1. ### 一句话区分
    

- **Monorepo**：
    

> - 一种**代码组织和工程管理方式**，解决“代码怎么放、怎么协作、怎么高效构建”。
>     

- **微前端**：
    

> - 一种**运行时架构方案**，解决“应用怎么拆、怎么独立开发和部署”。
>     

一个关心 **工程阶段**，一个关心 **浏览器运行阶段**。

---

2. ### Monorepo 是什么
    

Monorepo 指：

> 把多个前端应用、工具包、公共模块，**放在同一个 Git 仓库中统一管理**。

典型结构：

```Plain
repo/
├── packages/
│   ├── app-a
│   ├── app-b
│   ├── admin
│   ├── shared-apis
│   └── shared-utils
├── package.json
└── nx.json
```

**Monorepo 解决的问题**：

- 多项目代码如何统一管理
    
- 公共代码如何复用
    
- 依赖版本如何统一
    
- 构建、测试、CI 如何不爆炸
    

Monorepo **不关心**：

- 页面如何加载
    
- 是否独立部署
    
- 是否多个应用共存一个页面
    

---

3. ### 微前端是什么
    

微前端指：

> 把一个“巨大的前端系统”，拆成多个 **独立开发、独立部署、运行时组合** 的子应用。

核心关键词只有三个：

- 独立开发
    
- 独立部署
    
- 运行时集成
    

微前端解决的问题：

- 多团队并行开发
    
- 超大系统解耦
    
- 不同技术栈共存
    

微前端带来的成本：

- 运行时复杂度
    
- 性能损耗
    
- 状态通信复杂
    
- 调试成本高
    

---

4. ### 两者关系
    

> **Monorepo 和微前端不是对立关系，而是正交关系。**

|   |   |
|---|---|
|组合|是否合理|
|Monorepo + 非微前端|✅ 最常见|
|Monorepo + 微前端|✅ 大厂常见|
|多 repo + 微前端|⚠️ 可行但成本高|
|多 repo + 非微前端|❌ 最容易失控|

---

## 二、为什么需要 npm workspaces + Lerna + Nx

这套组合不是为了“高级”，而是为了解决 **多项目协作的现实问题**。

---

## 三、为什么要「依赖管理 & 包链接」

5. ### 没有包链接会发生什么？
    

假设一个应用要用公共接口包：

```TypeScript
import { getList } from '@shared/apis';
```

如果没有包链接，只能：

- 复制代码（灾难）
    
- 用相对路径（路径地狱）
    
- 发 npm 包（开发体验极差）
    

**包链接的本质**：

> 像用 npm 包一样使用 **本地源码**，但不复制、不发布。

---

6. ### 为什么要统一依赖管理
    

如果每个项目各自安装依赖：

- React 版本不一致
    
- node_modules 爆炸
    
- hook、context、样式问题频出
    

**统一依赖管理的本质**：

> 所有包遵循一套依赖规则，共用一套依赖树。

---

## 四、什么是「多包脚本 & 多包发布」

7. ### 多包脚本
    

问题场景：

- 有 5 个应用
    
- 每个都有 `dev / build / test`
    

如果没有多包脚本：

```Bash
cd app-a && npm run dev
cd app-b && npm run dev
cd admin && npm run dev
```

**多包脚本的意义**：

> 在根目录，一条命令，统一操作多个包。

---

8. ### 多包发布
    

当公共包被多个应用依赖时：

- 版本号必须一致
    
- 依赖关系必须同步
    
- 改动必须原子化
    

多包发布的本质是：

> 保证“多个包的变更，是一个整体行为”。

即使暂时不用，也必须具备这种能力。

---

## 五、什么是「依赖分析」

9. ### 核心问题
    

> 我改了一行代码，到底该跑哪些项目？

---

10. ### 没有依赖分析的后果
    

- 任意改动 → 全量 build
    
- 构建时间指数级增长
    

---

11. ### 依赖分析在做什么
    

> 自动回答：**“这次改动影响了谁？”**

例如：

- 改了 `shared-apis`
    
- 那么只影响依赖它的应用
    

---

## 六、什么是「构建缓存」

12. ### 没有缓存的问题
    

- 没改代码
    
- 每次 build 都重来一遍
    
- 时间和机器资源被白白浪费
    

---

13. ### 构建缓存的本质
    

> **输入不变，输出就不该重算。**

输入包括：

- 源码
    
- 配置
    
- 依赖版本
    
- 构建命令
    

只要一致，直接复用上一次结果。

---

## 七、把所有问题串起来

一个真实的多项目工程，必须解决 4 个问题：

1. **代码怎么互相用？** → 依赖管理 & 包链接
    
2. **怎么统一操作多个项目？** → 多包脚本 / 多包发布
    
3. **改动影响谁？** → 依赖分析
    
4. **没变的能不能别再跑？** → 构建缓存
    

---

## 八、工具只是分工，不是重点

- npm workspaces：解决依赖和包链接
    
- Lerna：解决多包操作和发布
    
- Nx：解决依赖分析和构建缓存
    

**工具不是目的，解决问题才是。**

---

## 九、最终一句话总结

> **Monorepo 的核心不是“高级”，而是“不崩”。** 它用依赖管理解决“怎么复用代码”， 用多包脚本解决“怎么操作工程”， 用依赖分析和缓存解决“怎么别白跑”。
